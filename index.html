一些碎碎念

发表于 2020-05-29

写下这篇分析是因为我这学期去蹭课了，听老师讲了很多关于镜头分析的东西。因为不用和必修的同学们一样交期末大作业，所以可以随心所欲写一些东西。如果GD和TOP能出首新歌，我也不至于自己营业，去分析以前的东西。写完这篇已经觉得像是重新认识了他们一次一样（接下来我会用权和崔来代指他们）。

ZUTTER是2015年GD和TOP合作的一首单曲，展现了他们独有的hip-pop风格。它的MV我已经看过无数次了，可是每次看完，总觉得好像明白了什么，但又感觉隔着一层纱，隐隐约约总有一种答案就在眼前，可是就差那么一点，有种怎么也摸不到核心的迷茫感。

可以确定的是，这个MV借鉴了《雌雄大盗》和《低俗小说》的元素，各种暗喻已经很明显了，之前也有粉丝各种分析非常精彩。但是我始终有种直觉，这个MV如此精彩，那么它应该是自成闭环的。也就是说，这个MV不单单只是在“老子有钱有品位，老子有亡命天涯的伴侣”这样空洞的喊话，它应该是讲述了一个完整的故事，只是我看不懂。

有一天又刷到这个MV的视频，点开看了一遍，突然发现了一个奇妙的切入点——“造型”。在这个MV里，他们两个人换过很多套造型，但很奇怪，有的造型只出现一次，有的却出现了好几次。于是我突然找到了一个思路，难道造型里藏着什么秘密吗？以此为切入点观察，仿佛打开了新世界的大门。原来他们五年前玩的梗，我现在才看懂。

首先mv里有个巨大的用时间和空间组成的障眼法。时间上是线性推进的，并没有倒叙或插叙，而空间上有两个地方，两个地方各代表不同的含义。为了方便理解，我要给每个造型赋予一个清晰的人格。

先说第一套造型+第一个人格，也就是运动服对应的“自信”人格。从最开始，两个人就都有“我最酷最帅最有钱”的基本认知，但这个人格开始时是背对着观众的，没有露出脸来。

然后镜头切换，两个落魄的人被捆在了椅子上，第二套造型+第二个人格出现了。这一段借鉴的就是《低俗小说》里的画面，所以这个人格，先借用电影的名字命名为“低俗”人格吧。这时，一个不露脸的PRETTYBOY出现，准备伤害他们两个，画面一闪而过。

接下来是第三套造型+第三个人格，帽衫搭配牛仔裤，是两个年轻人的造型，我把它命名为“年轻”人格。两个年轻人在一个通道里来回游走，其他仓库都被上了锁，只有最尽头的2088仓库是开着的，于是打开门，“年轻”人格进去了（寓意进入名利场）。

从这里开始，故事开始分成了两个视角，一个是权的游戏人间，另一个是崔的疯狂玩乐，里面都包含了大量的隐喻，但那些大多是个人的部分，按造型细分的话会有很多人格定义，为方便整理故事，我把他们统称为“有钱”人格。

注意看权的这件衣服，只出现了一次，只在一个场景里。旁边是女人，背后有沙滩，衣服上是红唇，直白点来说就是像这首歌表现出来的基本基调那样：纵情享乐。可是接下来这件衣服再也没有出现过，而在同样的场景里，崔出现了。

这个镜头，是进入2088仓库中之后，两人第一次碰面。两人都穿了一身花西装，在满是美女的沙滩边上开心地玩着剪刀石头布。这就是第四套造型+第四个人格，张扬的花西装，潇洒至极的人生状态，这个人格，就称为“得意”人格吧。有意思的事情是，这两个“得意”人格在沙滩上旁若无人的玩着，下一秒镜头切换回了“自信”人格，然后镜头又切换回沙滩美女，“得意”人格依然无视所有背景板，开心地玩着弹彼此额头这种幼稚又亲密的游戏。这一段都是“得意”人格和“自信”人格来回切换，只有他们两个人，其他一切都是背景。

​

下一段，另一个关键人格出现了，是第五套造型+第五个人格，雌雄大盗，从电影里也能得知，这个人格的关键词就是“破坏”了。而“破坏”人格出现后，又变成了“破坏”人格和“自信”人格不停闪回。

如果仔细去看一下MV，会发现这一段非常妙。这段是副歌，从中间被一分为二，前半部分崔在唱的时候，是“得意”+“自信”来回切换，同时也是“自信”人格第一次正面看向了观众。后半部分是权，他在开口的瞬间，一下子变成了“破坏”+“自信”来回切换。简单来说，就是一段副歌分成了两段，前半段得意、自信闪回，后半段破坏、自信闪回。这种安排是偶然还是故意的呢？我认为是故意的，因为下面剧情的发展马上就给出了解释。

接下来进入了崔的人格中，各种自恋炫耀愤怒，可以稍微细分一点。在崔自恋的过程中，插入了一个两人头对头睡着的画面。这套造型只出现在一个场景中，是第六套造型+第六个人格，两人睡得很安心，是一种彼此互为依靠的感觉，这个就称为“依赖”人格吧。然后又是崔的愤怒和抗议，从歌词大意也能看出来，中心思想就是“不要对我指手画脚，不懂规矩的人离我远点”。但这里要插播一个白衬衣的造型。这件白衬衣在BABY GOOD NIGHT里出现过，崔又把这件白衬衣穿到了ZUTTER里。

而这时一个关键点出现了，还记得之前的花西装造型吗？也就是“得意”人格，但是这个人格所在的场景切换了，从海滩边，变成了寒冷的冰室里。也正是这一幕，让我在看过无数次MV之后，突然有了从造型上找剧情脉络的灵感。从正常思路来看，这个镜头拍的是他俩被困在冷库里，那么造型的目的应该是想要突出他们很冷很惨。既然是这样，穿雌雄大盗那套西装不行，可是穿运动服那套不行吗？又或者，最开始的被绑在椅子上穿的那套破衣服也不行吗？这明显比花西装看上去更可怜。可为什么那一套破衣服只出现了一次，但花西装却出现在了两个场景中呢？

也就是从这里，我相信每套造型背后都是有隐喻的，那就是，两个曾在海滩边被环绕着的“得意”男人，后来一起被冷库困住了，从“得意”变成了“失意”。而且还通过造型暗示了，只能是从“春风得意”变成“寥落失意”，而不是“自信”变失意，也不是“破坏”变失意。接下来的发展则验证了这个想法，因为就在两人迷茫无助的时候，突然窗户打开了，暗示有人来救他们。 最开始我一直以为，是别人救了他们，但当我把各个造型用人格具象化之后，真正的答案却自动浮现了出来——救他们的不是别人，是他们自己。

为什么这样说呢？还是要先从正常思路来想。如果是路人，怎么什么镜头都没有呢？这是一个连客串的猪都有特写镜头和名字的MV，救他们的人不可能没有任何信息出现。整个MV里没有名字的是他们自己的各种造型，也就是各种人格。所以推理下来，那个打开窗子拯救“得意→失意”人格的，正是他们自己，而且据我推测，是雌雄大盗造型的“破坏”人格。

​

最开始出现在上图这个场景里的是“低俗”人格。现在的场景是，绑匪还在，被绑在了后面的椅子上，但“低俗”人格消失不见了。换句话说，“低俗”人格并没有反杀后拯救自己，而是“自信”人格出现，解决了绑匪，同化了仓库里的“低俗”人格。而且，这个绑架的密室并不在2088仓库里，是另一个“地方”。

再看“破坏”人格，“破坏”人格潇洒地推开大门，从2088仓库里走出来了。之前讲过，两个年轻人进入2088仓库，进入名利场，经历了“得意”、“失意”，然后“破坏”（隐喻无畏）出现，终结一切后潇洒离开。于是一切就很清晰了，“自信”人格救了被绑架的“低俗”人格，“破坏”人格救了困在冰室里的“失意”人格。

同时因为“自信”和“低俗”是固有的本性特征，所以不在仓库里，而在心灵密室中。但“得意→失意”和想要“破坏逃离”，都是一种人生经历，而且是在2088号仓库里经历的，于是青年进去，雌雄大盗出来。所以两个空间并行，同步推进故事，才是这个MV真正的框架。

从头整理一下，会发现只有三套造型曾出现在两个场景里。

1、运动服+“自信”：厕所里，绑匪密室里

2、花西装+“得意→失意”：海滩边，冷库里

3、雌雄大盗+“破坏”：走廊里，大街上

这三个人格共同交错之下，这个故事才终于完整了。

结合着MV能更明显的发现，“自信”人格的主导者是崔（每次出现运动服造型，都是崔的声音，而且自信人格的部分崔的动作更大，连舞都跳得更欢脱）。“破坏”人格的主导者是权（早年R.O.D里，他就用了雌雄大盗的梗。ZUTTER的MV里每次出现雌雄大盗造型，都是他的声音，然后他打开大门带着崔从爆炸的仓库开车离开）。而“得意→失意”的部分，两人都有唱，很平均（花西装第一次出现时是崔的声音，第二次出现时是权的声音）。理清了最关键的双人部分后，各自单人部分隐喻也就都清晰明确起来了。权穿着睡衣赌博赢了，寓意钱很多，存折里的“0”如同球桌上的球；往油画后面的保险箱里藏东西，寓意着他最在意的是艺术。崔养猪，给猪（猪在韩语里的谐音是钱）撒食物，寓意自己在持续“赚钱”，财富不停增长；崔的杀戮秀，寓意着我不好惹，不守规矩的人就是在自寻死路。

整体概括下来就是，开场是两个人天生自信，坚信“我超拽”。然后是故事的引子，“低俗”人格被绑架，接着被威胁、被伤害。此后故事正式开始，年轻的权和崔敲开2088仓库的大门，进入了名利场，开始追逐财富，追逐享乐。他们分开行动，经历了很多事情，但也一起得到了许多，一起春风得意。然而问题接连发生，两人被困在了冷库里，就连最后一点用来取暖的火都没了。这时“破坏”人格及时出现，把“失意”救了。这就是2088仓库里的故事线。 另一条线上，“低俗”人格部分也被拯救了（崔主导的自信人格把绑匪绑了起来），最终雌雄大盗冲出2088仓库，浪迹天涯（寓意自由），自信人格接手低俗人格。同时也正因为“自信”人格和“低俗”人格一开始就不在2088仓库里，是他们的本我部分，所以“破坏”人格炸掉仓库后，“自信”人格毫发无伤，还在开心地跳舞。

对应的歌词也很有意思，“自信”人格在密室里跳舞的时候，是很完整的一段RAP词，“这是GD和大吉的合作”，中途没有任何场景和画面的切换，很显然，GD和大吉是他们事业上的代表作。这都是在从不同角度诠释他们的本质人格，而这个诠释，又是一定要配对。

最终不管是在本我认知上，还是人生经历上，两个人都依然在一起。MV的最后，他们改了雌雄大盗原型的结局，变成开车浪迹天涯，标准的happy ending。一个完整的闭环，在ZUTTER的MV里完美成型。

【下面这段话本来想放在最前面，但怕影响阅读体验，就放在了最后。】

我之前找过很多ZUTTER的分析，在没怎么细究的情况下，好像大家一般认为这个MV讲述了两个人过于自信和享乐，被绑架，然后又自己跑出来的故事。可我一直觉得这个故事线不完整，前后有好几个地方对不上。MV里虽然出现了很多造型，但造型的出现都是有意义的，这条单一的故事线并不能解释它们以及《雌雄大盗》和《低俗小说》出现的意义。所以我一直对这个MV念念不忘，直到今天，也算是拼出了一个逻辑线清晰的故事。

下面紧接着就是另一个问题了，那就是：他们本人有没有想这么多呢？又或者，MV是导演拍的，代表的是导演的意图。但是我想说，凡是艺术作品，导演和表演者彼此之间，一定是要有绝对信任的，不然出来的东西就会驴唇不对马嘴，一团混乱。毫无疑问，ZUTTER的MV非常有逻辑，而且真正看懂了之后就会发现，它是没有漏洞、自成闭环的。里面借用的各种电影元素和隐喻意象，也全是崔和权的审美取向，如果全是导演的意思，那导演要对他们两个人有多深入的了解才能达到这种程度呢？再看造型上，崔拿出了多年前BABY GOOD NIGHT时候的衣服，后面权发INS宣传的时候，还把ZUTTER的打歌服拍了一张断背山风格的照片，衣服有什么含义，我不信他们不懂。

这个MV当然可以用很正常的思路去看，从头到尾就是讲了“我超有钱超帅”。可是一上来就是《低俗小说》，后面又跟着《雌雄大盗》，但凡深入一点，就会发现这里面的私货和暗号太多了，多到没办法推给造型师、导演、道具师、摄影师……那么排除掉种种“不可能”之后，就只剩下一种可能，他们两个知道自己在拍什么，有足够的默契，最终作品成型，就像这首歌的名字一样，ZUTTER！

很多时候，可能分析的人最被吐槽的点就是想太多，但就像老师讲过的那样，所谓的想太多，很多是建立在原著的空白上，所以不管多夸张的脑洞都能塞进去。然而这个MV里，一切都已经表达出来了，低俗小说和雌雄大盗明晃晃摆在那里了，只不过是看粉丝能不能看懂而已。所以面对着这两个喜欢暗戳戳的人，我只觉得，我想得是不是还不够多？不然为什么一个MV里，就有这么多严丝合缝的“巧合”呢？如果这个作品没有那么多的寓意，为什么每个点都能对上呢？甚至就连两人rap的段落，也能和造型，故事线对上。这么多巧合只能说明，是他们有意制造的。

VHDL概述

VHDL的程序结构

一个VHDL程序对应着一项具体工程设置，通常被称为设计实体。设计实体一般由库，程序包，实体，结构体，配置五个部分组成。实体和结构体是一个设计实体中必须包含的两个基本组成部分。

实体

实体用于描述设计实体与外部的接口信号，它从外观上描述了一个设计实体，规定了其输入输出接口信号，是设计实体对外的一个通信界面

实体说明一般具有以下语句结构：

类属参数说明

类属参数说明语句由关键词GENERIC引导一个类属参数表，表中提供设计实体和外部环境通信的各种静态信息

格式

端口说明

端口说明语句是对一个设计实体界面的说明。端口为设计实体和外部坏境的动态通信提供通道

格式

端口模式 模式说明
IN 输入，只读模式，主要用于时钟输入，控制输入及单向的数据输入
OUT 输出，单向赋值模式，常用于计数器输出
BUFFER 具有读功能的输出模式
INOUT 信号是双向的，既可以进入实体，也可以离开实体

结构体的一般语句格式

结构体说明语句

结构体说明语句用于对结构体内部使用的信号，常数，数据类型，元件，函数和过程等进行说明，不是必须的，有时可以省略

功能描述语句结构

它含有5种不同类型，以并行方式工作的语句结构，分别是：块语句，进程语句，子程序调用语句，信号赋值语句，元件例化语句。

库

库是一种用来存储预先完成的程序包和数据集合体的仓库。库的说明总是放在设计实体的最前面，格式如下：

LIARARY.库名

常用的库 库的说明
IEEE 最常见的库
STD 标准程序包STANDARD,TEXTIO都包含在STD库中
WORK 用于存放用户设计和定义的一些设计单元和程序包
VITAL 可以提高VHDL门级时序模拟的精度
程序包

程序包是构成库的单元，一个库中可以包含多个不同的程序包。通过将已说明或定义的信号，常数，数据类型，元件，函数和过程等收集到一个程序包中，并将它们并入一个库中，可以使它们适应一般额访问和调用范围。

VHDL 的描述风格
  • 行为描述：只考虑电路输入与输出之间的转换行为，而不关心实现预定功能的硬件结构
  • 数据流描述：数据从输入到输出流动
  • 结构描述：对设计单元的硬件结构进行描述

VHDL的语言要素

文字规则
数字型文字
  1. 整型文字：由数字和下划线组成
  2. 实数文字：带有小数点的一种十进制的数
  3. 以数制基数表示的文字：例如10#36#2 表示3600
  4. 物理量文字：如50s,100m等
字符串型文字
  1. 文字字符串：放在双引号中的一串文字，如“RIGHT”
  2. 数字字符串：如B’ ‘110110010 代表二进制数数组，位矢量长度为9
  3. 字符：放在单引号里的ASCII字符，如‘R’，‘6’
标识符

标识符用来定义常数，变量，信号，端口，子程序，或参数的名字。VHDL的基本标识符是以英文字母开头，不连续使用下划线，不以下划线结尾的，由26个英文大小写字母，数字0~9及下划线“_”组成的字符串

下标名及下标段名

下标名用于指示数组型变量或信号的某一元素，下标段名用于指示数组型变量或信号的某一段元素

关系的形式化定义及有关概念

域 ：域是一组具有相同数据类型的值的集合，又称为值域。整数，实数和字符串的集合都是域。域中所包含的值的个数称为域的基数。

笛卡尔积：给定一组域D1,D2…Dn(它们可以包含相同的元素，即可以完全不同，也可以部分或全部相同)，D1.D2…Dn的笛卡尔积为：

笛卡尔积实际上是一个二维表

关系：笛卡尔积D1XD2X…Dn的任一子集称为定义在域D1,D2….Dn上的n元关系

在关系模型中，关系可以进一步定义成关系头+关系体

关系的性质

关系是一种规范化了的二维表中行的集合

  • 每一列中的分量必须来自同一个域，必须是同一类型的数据
  • 不同的属性可以来自同一个域，不同的属性必须有不同的名字
  • 列的顺序可以任意交换
  • 关系中元组的顺序可以任意
  • 关系中不允许出现相同的元组
  • 关系中每一分量必须是不可分的数据项
关系模式

关系的描述称为关系模式

Google

在我对谷歌的力量一无所知的时候，天真地以为它就是一个普通的搜索引擎，直到我学会了一些“黑魔法”，才体会到它的方便好用。

  • OR, AND

OR: 返回的结果是包含OR两边的任意关键词，比如: amazon OR ebay

AND: 返回的结果是包含AND两边的关键词，比如: amazon AND ebay

注意：OR, AND 必须是大写

  • 用 “-“ 在搜索结果中排除某个关键词。比如搜索 amazon -ebay，搜索的结果就不会出现ebay。
  • 用 * 星号搜索指定关键词

在Google中搜索 “amazon * guide”，其中的*号代表任何文字。返回的结果就不仅包含 Amazon Quick Start Guide 还包括 Amazon PPC Guide

  • intext: 指返回的页面正文中包含关键词的页面。

例如：在google中搜索 “intext:amazon ebay wish” 就会返回页面正文中包含 “amazon ebay wish”的页面

  • allintitle：搜索返回的是页面标题中包含多组关键词的文件。

例如：allintitle:amazon ebay wish

就相当于:intitle:amazon intitle:ebay intitle:wish

返回的是标题中既包含“amazon”, “ebay” , “wish” 的页面。

  • filetype:

基本查询语法：[关键字1]+ [空格]+[filetype]+[:]+[文件类型标识]　　

filetype:指令用来搜索特定的资源，比如PDF电子书、Word文件等非常有用。

例子: SEO filetype:pdf 返回的就是包含SEO这个关键词的所有 PDF文件。

  • related:

查询与某个网站相关联或者相类似的页面

例如搜索：related:amazon.com，就得出了 Rakuten 等电商平台

Stack over flow

我强烈建议学会使用，并习惯Stackoverflow.。这是个英文网站，很多问题和回答都是英文的，因此最好养成用英文来搜索问题的习惯。想要得到准确的解决方案的前提是能精确定位你的问题，我把它简单概括为三步。第一步：你使用的编程语言是什么.永远在输入问题关键词的时候先输入你使用的是哪种编程语言。第二步：定位你在做什么事情时遇到了什么问题。比如我曾经在使用Anaconda时遇到了安装包装不上的问题，就可以这样描述：Anaconda install package.第三步：复制错误信息。有了这三步，只要你不是第一个遇见某个技术问题的人，你基本就会在 stackoverflow 找到相同或者类似的问题。

在Stackoverflow上也是可以提问的，但是我建议提问前一定要搜索，当实在找不到答案后再提问。对于那些靠搜索就能得到答案的问题，StackOverFlow 上的广大程序员是毫不留情的，会让你的反对票快速上升。。。如果你问了一些低质量的提问，是会被关闭或者删除的。既然讲到提问，还是要学习一些典型的提问/回答习惯，这里提供一些笔记

提问：
1，In my limited experience with .. 谦虚表达自己在某方面的经验

2， I am searching for a long time on net. But no use. Please help or try to give some ideas how to achieve this. 找了很久未果，求助

3，after searching around for a decent XX solution and found that everything out there was difficult to use. 找了N种方法都发现太难了。

4，I’ ve looked around and apparently I’ve got the choice between these libraries/ solutions: 说明自己是努力搜索过的，然后目前有了哪几种方案

5，which seems it’s a promising solution. 看起来是一个好解决方案

6,I haveve tried multiple variations of this, but none of them seem to work. Any ideas? 试了很多种方法都无效，求助

7，Wanted to know if it’s good practice to do that and what would be the best way to do that? 我的做法是否正确，是否有更好的方法？

稍微提一句，在它的官方说明里有一段话：Please do not add a comment on your question or on an answer to say “Thank you”. Comments are meant for requesting clarification, leaving constructive criticism, or adding relevant but minor additional information – not for socializing. If you want to say “thank you,” vote on or accept that person’s answer, or simply pay it forward by providing a great answer to someone else’s question.也就是说在提问的时候不要写“Thank you”,你的点赞就是对回答者的感谢。

之所以写出如何提问，是因为之前老师讲过的提问的智慧让我印象很深刻。当你拋出一个技术问题时，最终是否能得到有用的回答，往往取决于你所提问和追问的方式。 有时候，清晰描述一个问题，特别是技术问题没有想象的那么简单。提问从来就是一门学问，可惜很多人没有意识到这一点，或者没有给予足够的重视。

数据库原理与应用（一）

发表于 2020-03-20 更新于 2020-03-25

1.1 信息 数据 数据处理与数据管理**

数据

* 数据是信息的具体表现形式

* 数据的表现形式：数字，文字，图形，图像，声音等

数据与信息的联系

* 数据是信息的符号表示

* 信息是数据的内涵，是对数据的语义解释

数据处理与数据管理

* 数据处理：将数据转换成信息的过程，包括数据的收集，管理，加工利用乃至信息演变推导

* 数据管理：数据的分类，组织，编码，存储，维护，检索等

信息=数据+数据处理

数据处理的真正含义应该是为了产生信息而处理数据

信息=数据+语义

1.2 数据库技术的产生，发展**

  1. 人工管理阶段：数据不共享，无独立性
  1. 文件系统阶段：程序与数据间有一定独立性，数据具有一定共享性
  1. 数据库系统阶段：数据结构化，共享性高，独立性高，有统一的数据管理和控制功能(DBMS)

​ DBMS: 数据的安全性控制，完整性控制，并发控制，数据恢复

1.3 数据库系统（DBS）的组成**

* 数据库：数据库是存储在计算机内，有组织的，可共享的数据和数据对象集合

* 用户：对数据库进行存储，维护，检索等操作。包括最终用户，应用程序员，数据库管理员

* 软件系统：DBMS,OS,应用开发工具，应用系统

* 硬件系统

1.4 数据库系统的内部体系结构**

数据库系统内部的体系结构从逻辑上分为三级：外模式 模式 内模式

* 外模式：数据库用户能看到并允许使用的那部分局部数据的逻辑结构和特征的描述 （它是逻辑模式的某一部分的存储表示）

* 模式：数据库中全体数据的逻辑结构和特征的描述 （它是整个数据库实际存储的抽象表示）

主要编码方案

最早的字符串编码是 ASCII 编码，它仅仅对 10 个数字、26 个大小写英文字母以及一些特殊字符进行了编码。ASCII 码最多只能表示 256 个符号，每个字符只需要占用 1 个字节。

随着信息技术的发展，各国的文字都需要进行编码，于是相继出现了 GBK、GB2312、UTF-8 编码等，其中 GBK 和 GB2312 是我国制定的中文编码标准，规定英文字符占用 1 个字节，中文字符占用 2 个字节；而 UTF-8 是国际通过的编码格式，它包含了全世界所有国家需要用到的字符，其规定英文字符占用 1 个字节，中文字符占用 3 个字节。

  • 汉字字符集

  1. GBK (兼容GB2312)
    GBK：不再要求低字节一定是127号之后的编码，只要第一个字节是大于127就固定表示这是一个汉字的开始，不管后面跟的是不是扩展字符集里的内容

  2. GB2312 (分级，排序，2个字节，编码表)

GB2312：一个小于127的字符意义与原来的相同，但是两个大于127的字符连在一起时，就表示一个汉字；前面的一个字节（高字节）从0xA1用到0xF7，后面一个字节（低字节）从0xA1到0xFE；其中，在ASCII里本来就有的数字、标点、字母都统统重新编了两个字节长的编码，这就是常说的“全角”字符，而原来在127号以下的那些就叫“半角”字符

GB2312共收录6763个汉字，其中一级汉字3755个，二级汉字3008个；同时收录包括拉丁字母、希腊字母、日文平假名及片假名字母、俄罗斯语西里尔字母在内的682个全形字符。

GB2312区位码

GB2312对所收汉字进行了“分区”处理，每区含有94个汉字/符号。这种表示方式也称为区位码。

01-09区为特殊符号。
16-55区为一级汉字，按拼音排序。
56-87区为二级汉字，按部首/笔画排序。
10-15区及88-94区则未有编码。

举例来说，“啊”字是GB2312之中的第一个汉字，它的区位码就是1601。字节编码，通常采用EUC储存方法，以便兼容于ASCII。每个汉字及符号以两个字节来表示 。 第一个字节称为“高位字节”，第二个字节称为“低位字节”。 “高位字节”使用了0xA1-0xF7(把01-87区的区号加上0xA0)，“低位字节”使用了0xA1-0xFE(把01-94加上0xA0)。例如 “啊”字在大多数程序中，会以0xB0A1储存（与区位码对比：0xB0=0xA0+16,0xA1=0xA0+1）。

所以GB2312编码中汉字区码的十进制是从176到247，位码是从161到255.之所以存储了6763小于82*94=6768，是因为在区码为215，位码为250-254之间共五个编码没有汉字编码，所以6768-5=6763个。

GB2312编码规则

2字节编码，高位为0xA1-0xF7，低位为0xA1-0xFE

汉字区域，高位为0xB0-0xF7，低位为0xA1-0xFE

特殊符号，高位为0xA1-0xA9，低位为0xA1-0xFE

读取文件时的编码问题

在 Python 中，有 2 种常用的字符串类型，分别为 str 和 bytes 类型，其中 str 用来表示 Unicode 字符，bytes 用来表示二进制数据。str 类型和 bytes 类型之间就需要使用 encode() 和 decode() 方法进行转换。Python 3.x 默认采用 UTF-8 编码格式，有效地解决了中文乱码的问题。

Python encode()方法

encode() 方法为字符串类型（str）提供的方法，用于将 str 类型转换成 bytes 类型，这个过程也称为“编码”。

encode() 方法的语法格式如下： 

str.encode([encoding="utf-8"][,errors="strict"])

注意，格式中用 [] 括起来的参数为可选参数，也就是说，在使用此方法时，可以使用 [] 中的参数，也可以不使用。

参数 含义
str 表示要进行转换的字符串。
encoding = “utf-8” 指定进行编码时采用的字符编码，该选项默认采用 utf-8 编码。例如，如果想使用简体中文，可以设置 gb2312。当方法中只使用这一个参数时，可以省略前边的“encoding=”，直接写编码格式，例如 str.encode(“UTF-8”)。
errors = “strict” 指定错误处理方式，其可选择值可以是：
             strict：遇到非法字符就抛出异常。

             ignore：忽略非法字符。

             replace：用“？”替换非法字符。

             xmlcharrefreplace：使用 xml 的字符引用。该参数的默认值             为strict。 |

注意，使用 encode() 方法对原字符串进行编码，不会直接修改原字符串，如果想修改原字符串，需要重新赋值。

【例 】将 str 类型字符串“崔胜铉什么时候出solo”转换成 bytes 类型。 

str ="崔胜铉什么时候出solo"
str.encode("gbk")

【输出结果】b’\xb4\xde\xca\xa4\xee\xe7\xca\xb2\xc3\xb4\xca\xb1\xba\xf2\xb3\xf6solo’

Python decode()方法

和 encode() 方法正好相反，decode() 方法用于将 bytes 类型的二进制数据转换为 str 类型，这个过程也称为“解码”。

decode() 方法的语法格式如下：

bytes.decode([encoding="utf-8"][,errors="strict"])
参数 含义
bytes 表示要进行转换的二进制数据。
encoding=”utf-8” 指定解码时采用的字符编码，默认采用 utf-8 格式。当方法中只使用这一个参数时，可以省略“encoding=”，直接写编码方式即可。注意，对 bytes 类型数据解码，要选择和当初编码时一样的格式。
errors = “strict” 指定错误处理方式，其可选择值可以是：strict：遇到非法字符就抛出异常。ignore：忽略非法字符。replace：用“？”替换非法字符。xmlcharrefreplace：使用 xml 的字符引用。该参数的默认值为 strict。

统计汉字字频

思路：是先把每个字符提出来放在列表里；再过滤掉其中的标点符号；最后用字典对某个字出现的频率进行累加。 

#coding:utf-8

word_lst = []
word_dict = {}

exclude_str = "，。！？、（）【】<>《》=：+-*—“”…" 

with open("d:/nlp/EX/top.txt","r") as fileIn ,open("d:/nlp/EX/字频.txt",'w') as fileOut:

# 添加每一个字到列表中
for line in fileIn:
    for char in line:
        word_lst.append(char)

# 用字典统计每个字出现的个数       
for char in word_lst:
    if char not in exclude_str:
        if char.strip() not in word_dict: # strip去除各种空白
            word_dict[char] = 1
        else :
            word_dict[char] += 1

# 排序
#   x[1]是按字频排序，x[0]则是按字排序
lstWords = sorted(word_dict.items(), key=lambda x:x[1],  reverse=True) 

# 输出结果 (前100)
print ('字符\t字频')
print ('=============')
for e in lstWords[:100]:
    print ('%s\t%d' % e)
    fileOut.write('%s, %d\n' % e)
